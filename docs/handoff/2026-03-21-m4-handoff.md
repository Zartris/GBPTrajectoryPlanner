## What's Broken — THE CRITICAL BUG

### Jacobian sign inversion + arc-length Jacobian on different trajectories

**Location:** `crates/gbp-agent/src/robot_agent.rs`, function `update_interrobot_jacobians`, lines ~240-260

```rust
let diff = s_a - s_b;
let dist = diff.abs();
let sign = if diff >= 0.0 { 1.0 } else { -1.0 };
irf.jacobian_a = -sign; // negative because residual = d_safe - dist
irf.jacobian_b = sign;
irf.dist = dist;
```

**Problem 1 — Jacobian sign is inverted:**
The residual is `d_safe - dist`. When `dist < d_safe` (too close), residual is positive. The Jacobian `jacobian_a = -sign(s_a - s_b)` pushes the AHEAD robot BACKWARD instead of forward. This causes the front robot to stop/stutter even though it should accelerate away.

The correct Jacobian for "push apart":
- Robot ahead (s_a > s_b): jacobian should push s_a UP (positive) → `jacobian_a = +1`
- Robot behind (s_a < s_b): jacobian should push s_a DOWN (not applicable since velocity ≥ 0)

But this sign analysis assumes `residual = d_safe - dist` where positive means "violation, push apart". The Schur complement and information form math need careful rederivation.

**Problem 2 — Arc-length distance is meaningless on different trajectories:**
`dist = |s_a - s_b|` where s_a and s_b are global arc-lengths on DIFFERENT trajectories. Robot A at s=30 on trajectory 1 and Robot B at s=30 on trajectory 2 could be meters apart or overlapping. The factor has no idea.

The activation check correctly uses 3D distance (fixed earlier), but the Jacobian and residual still use arc-length.

**The fix discussed but not implemented:**
Use 3D Jacobian: `d(dist_3d)/d(s_a) = (pos_a - pos_b) · tangent_a / dist_3d`

This requires:
- `pos_3d_a`, `pos_3d_b` — 3D positions at this timestep k
- `tangent_a` — unit tangent of robot A's trajectory at this position
- Computing these in `update_interrobot_jacobians` using the trajectory + map

### GBP velocity oscillation (partially fixed)

The exponential decay precision (replacing the hard hinge cutoff) reduced oscillation significantly. The endcollision scenario works well — smooth deceleration, stable 1.300m clearance at goal.

The merge scenario still has issues because of the Jacobian bug above — robots pass through each other and swap positions.

### DynamicConstraints lag

The jerk limiter takes ~0.5s to decelerate from 2.5 m/s to 0. At 2.5 m/s, the robot covers ~0.5m during deceleration — comparable to the old d_safe. With the new d_safe=1.3m this is less critical but still means the GBP output should start decelerating well before d_safe.

The robotics expert review (see below) noted this as concern I1.

## What Needs to Happen Next

### 1. Fix the Jacobian (CRITICAL — blocks M4 merge)

Replace arc-length Jacobian with 3D Jacobian in `update_interrobot_jacobians`:

```rust
// Instead of:
let diff = s_a - s_b;
let dist = diff.abs();
let sign = if diff >= 0.0 { 1.0 } else { -1.0 };
irf.jacobian_a = -sign;

// Use 3D:
let pos_a = /* eval_position for variable k on our trajectory */;
let pos_b = /* eval_position for variable k on their trajectory (from broadcast) */;
let tangent_a = /* eval_tangent for variable k on our trajectory */;
let separation = [pos_a[0]-pos_b[0], pos_a[1]-pos_b[1], pos_a[2]-pos_b[2]];
let dist_3d = sqrt(separation dot separation);
let jacobian_a = (separation dot tangent_a) / dist_3d; // scalar: d(dist_3d)/d(s_a)
irf.jacobian_a = jacobian_a;
irf.jacobian_b = 0.0; // B is marginalized, its Jacobian is handled by ext_belief
irf.dist = dist_3d;
```

This needs:
- `map.eval_position(edge, local_s)` for each variable k — already done for activation
- `map.eval_tangent(edge, local_s)` for robot A's trajectory at variable k
- The `dist` field set to 3D distance, not arc-length
- The `_map` parameter in `update_interrobot_jacobians` changed from `_map` to `map`

### 2. Add CBF safety layer (replaces removed last_max_position)

The `last_max_position` hard clamp was removed because it froze both robots. A proper Control Barrier Function should:
- Only constrain velocity, not position
- Be directional (only slow down if approaching collision, not if moving away)
- Work in 3D (not arc-length)
- Be a separate module like `DynamicConstraints`
